## 1. Webtop 실행 방법

### 사전 준비

- **Docker** 설치
- 터미널(Mac/Linux) 또는 PowerShell(Windows)을 실행.
- 포트 `3000`, `3001`이 다른 프로그램에 사용 중이지 않은지 확인.

```bash
# 포트 사용 여부 확인 (Mac/Linux)
lsof -i :3000
lsof -i :3001
```


### Step 1 · 컨테이너 실행

```bash
docker run -d \
  --name=webtop \
  -e PUID=1000 \
  -e PGID=1000 \
  -e TZ=Etc/UTC \
  -p 3000:3000 \
  -p 3001:3001 \
  --shm-size="1gb" \
  --restart unless-stopped \
  lscr.io/linuxserver/webtop:latest
```

**확인:** 아래 명령어 실행 후 `webtop` 컨테이너가 `Up` 상태인지 확인

```bash
docker ps
```


### Step 2 · 브라우저로 접속

컨테이너가 뜨고 나서 브라우저에서 http://localhost:3000 으로 접속.

**확인:** 브라우저 안에 **리눅스 데스크탑 화면**이 표시되면 정상.


### Step 3 · 기능 동작 확인

- **원격 브라우저 실행**: 바탕화면의 브라우저 아이콘 클릭, 원격 컨테이너 안의 브라우저가 열림
- **인터넷 접속**: 원격 브라우저에서 `https://google.com` 접속, 구글 홈페이지가 로드됨
- **클립보드 동기화**: 로컬에서 텍스트 복사 → 원격 브라우저 주소창에 붙여넣기, 텍스트가 그대로 입력됨
- **한글 입력**: 원격 브라우저 주소창에 한글 입력, 한글이 깨지지 않고 정상 입력됨
- **세션 유지**: 브라우저 탭을 닫고 다시 `localhost:3000` 접속, 이전 상태가 그대로 유지됨


### Step 4 · 종료 및 정리

```bash
# 컨테이너 중지
docker stop webtop

# 컨테이너 삭제
docker rm webtop
```

## 2. 나만의 RBI 시스템 — 기능 요구사항

Webtop을 써보고 나서 저만의 RBI 시스템을 만들 때 구현하고 싶은 기능들입니다. 다 할 수 있을진 모르겠지만 최대한 많이 적어봤습니다.

#### 파일 전송
- 드래그 앤 드롭 방식의 업로드/다운로드 인터페이스

#### 인증 및 세션 관리
- 사용자 인증(Auth) 처리
- 사용자별 독립 컨테이너 세션 할당

#### 컨테이너 오케스트레이션
- **K8s 연동:** 다수 사용자를 위한 컨테이너 자동 스케일링 및 관리

#### 관리자 대시보드 / 정책 관리 페이지
- 사용자별 권한 제어 (예: 파일 전송 비활성화, 특정 URL 접근 차단)
- 정책 템플릿 및 그룹 단위 설정 지원

#### 화면 워터마킹
- 화면 캡처를 통한 정보 유출을 방지하기 위해 사용자 IP나 아이디같은 것을 투명하게 표시

#### 가상 프린팅
- 원격 인쇄 리다이렉션: 원격 컨테이너 내의 인쇄 명령을 가상 프린터 드라이버로 캡처하여 PDF 파일로 변환 후 로컬로 전송
- 출력물 보안 제어: 출력 시 워터마크 강제 삽입 및 인쇄 이력(누가, 언제, 무엇을)에 대한 상세 로그 저장

#### 화면 녹화
- 보안 사고 발생 시 추적을 위해 브라우저를 녹화하거나 모든 클릭/입력 로그를 저장

#### 화상회의 지원 (WebRTC)
- 로컬의 카메라 및 마이크 스트림을 원격 컨테이너로 전달하여 브라우저 내 화상회의 가능하도록 구현

## 3. Webtop vs Selkies — 비교 분석

### 개요

| 항목 | Webtop | Selkies |
|------|--------|---------|
| 기반 기술 | KasmVNC (VNC over WebSocket) | WebRTC + GStreamer |
| 스트리밍 방식 | VNC 프로토콜 → WebSocket 변환 | WebRTC P2P / 미디어 스트리밍 |
| 지연시간 | 보통 (50~150ms) | 낮음 (10~50ms, LAN 기준) |
| 화질 / FPS | 보통 (정적 화면 최적화) | 높음 (동영상, 애니메이션에 강함) |
| 초기 설정 난이도 | 쉬움 (Docker 한 줄 실행) | 복잡 (WebRTC, TURN 서버 필요) |
| 유지보수 난이도 | 낮음 (LinuxServer.io 공식 지원) | 높음 (직접 빌드/설정 필요) |
| 컨테이너 크기 | 보통 (~1GB) | 무거움 (GPU 드라이버 포함 시) |
| GPU 가속 | 미지원 | 지원 (NVIDIA/AMD) |


### Webtop 장단점

**장점**
- 빠른 시작: Docker 명령 하나로 즉시 실행, 별도 인프라 불필요
- 안정적인 세션 유지: VNC 특성상 네트워크 일시 단절 후에도 세션 복구 용이
- 한글 입력 안정적: IME 처리가 비교적 안정적
- 경량성: GPU 없이도 동작, 서버 비용 절감

**단점**
- 높은 지연시간: VNC 특성상 Selkies 대비 체감 응답속도 느림
- 화상회의 제약: 카메라/마이크를 WebRTC로 원격 컨테이너에 전달하기 어려움
- 화질 한계: 동영상 재생, 고FPS 작업에서 화질 저하 발생
- GPU 미지원: 그래픽 집약적 작업에 부적합


### Selkies 장단점

**장점**
- 낮은 지연시간: WebRTC 기반으로 실시간 인터랙션에 최적화
- 고화질 스트리밍: 동영상, 애니메이션, GPU 작업에서 뛰어난 성능
- 화상회의 네이티브 지원: WebRTC 설계 특성상 카메라/마이크 전달이 자연스러움
- GPU 가속: NVIDIA, AMD GPU 활용 가능

**단점**
- 높은 초기 설정 복잡도: TURN/STUN 서버, WebRTC 설정, GStreamer 파이프라인 구성 등 필요
- 운영 부담: 공식 관리 이미지 부재, 직접 빌드 및 유지보수 필요
- 문서/커뮤니티 부족: 한국어 자료 거의 없음, 트러블슈팅 어려움
- 무거운 이미지: GPU 드라이버 포함 시 이미지 크기 및 리소스 소비 증가


## 4. 결론 — Webtop으로 시작하는 이유

프로젝트를 시작하고 기능들을 추가할 때 더 Webtop을 이용하면 훨씬 수월하고 편리하게 개발할 수 있을 것 같아서 골랐습니다. 

Docker 한 줄 명령으로 완전한 원격 데스크탑 환경이 동작합니다. Selkies는 TURN 서버, WebRTC 설정 등 인프라 구성에만 수일이 소요될 수 있으나, Webtop은 당일 내 핵심 RBI 기능 검증이 가능합니다.

추가하고 싶은 기능들(파일 전송, 인증, 관리자 대시보드, 워터마킹, 화면 녹화, 가상 프린팅)을 Webtop 기반에서 충분히 구현 가능하다고 생각했습니다. 화상회의 기능은 꽤나 어려울 것으로 예상되지만 한번 시도해보고 싶습니다. 

Webtop으로 MVP를 완성한 후, 실제 테스트를 해보고 지연시간이나 화상회의가 병목이 될 시점에 Selkies 전환을 생각해봐도 될 것 같습니다. 

### 남은 계획

| 단계 | 목표 | 할 일 |
|------|------|-----------|
| 1 | Webtop 기반 MVP | 인증, 세션 관리, 파일 전송, 관리자 대시보드 구현 |
| 2 | 보안 기능 강화 | 워터마킹, 화면 녹화, 가상 프린팅, K8s 연동 |
| 3 | 성능 고도화 검토 | 지연 속도 최적화, Selkies 전환? |
